# Maze Generator and Solver

This project implements a dynamic maze generator and solver using Pygame and OpenGL. It fulfills all requirements of the assignment, including the **Bonus/Challenge** features.

## How it Works

### 1. Maze Generation (The "Mouse" Logic)
The maze is generated using a **Stack-based Depth First Search (DFS)** algorithm:
- We start with a grid where all walls are intact.
- A "mouse" starts at a random cell and moves to an unvisited neighbor, "eating" the wall between them.
- If multiple unvisited neighbors exist, one is chosen randomly, and the others are stored on a **Stack**.
- When the mouse reaches a dead end, it backtracks by popping the stack until it finds a cell with unvisited neighbors.
- **Bonus Feature**: There is a 5% (1 in 20) chance at each step that the mouse eats an *extra* wall. This creates **cycles** in the maze, making it non-planar and breaking the simple "shoulder-to-the-wall" rule.

### 2. Maze Solving (Backtracking)
The solver uses a backtracking algorithm to find the path from the start to the end:
- **Red Dots**: Represent the current path being explored.
- **Blue Dots**: Represent dead ends that the solver has backtracked from.
- **Bonus Feature**: The starting (Green) and ending (Purple) cells are placed in the **interior** of the maze, rather than just on the edges.

## Data Structure
As required, the maze is represented by two primary 2D arrays:
- `northWall[R][C]`: 1 if the north wall exists, 0 otherwise.
- `eastWall[R][C]`: 1 if the east wall exists, 0 otherwise.

## Theoretical Questions

**Q: Might a queue be better than a stack for storing candidates? How would it affect the order in which later paths are created?**
A: A **Queue** would implement a **Breadth-First Search (BFS)** approach. While it would still visit every cell, the resulting maze would look very different. Instead of the long, winding, "tortuous" paths generated by a Stack (DFS), a Queue would create a maze that radiates outwards from the start point in a more uniform, "square" fashion. This results in many short branches and a less challenging, more predictable maze structure.

**Q: Why do starting and ending cells on outer boundaries always allow finding the end even with cycles?**
A: If both points are on the outer boundary, you can always find the exit by following the "shoulder-to-the-wall" rule because the boundary itself forms a single continuous circuit. Even if there are internal cycles (islands of walls), the outer boundary wall remains a connected path that eventually leads from any boundary point A to any boundary point B. When start/end points are in the **interior**, cycles can trap the solver in a loop or force it to cross "islands," making simple wall-following insufficient.
